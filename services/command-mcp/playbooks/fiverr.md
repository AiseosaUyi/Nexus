# Fiverr — playbook (kind: inbound)

Fiverr is gig-based: buyers come to your **gigs**, message you, or post **Buyer Requests/Briefs**.
Less active for some, but don't leave the net closed. Response time is heavily weighted here.

## Read (each run)
- Inbox → buyer messages. FAST response matters for ranking. Scam-filter each.
- Briefs / buyer requests → matching asks worth an offer.
- Active orders → any needing delivery/update (flag to Aise; don't deliver files yourself).

## Draft
- Buyer messages: quick, clear, answer the question, guide toward a custom offer/scope.
- Briefs: concise pitch, reference the ask, propose package + timeline.
- Keep it on-platform; scam-filter "contact me on WhatsApp / pay outside Fiverr" (against ToS + risky).

## What good looks like on Fiverr
- Sharp gig titles, strong gig thumbnails (first frame = the sell), clear packages/pricing,
  fast response badge, samples in the gallery.
- Niche gigs ("SaaS landing page UI") outperform generic ("I will design anything").

## Health inputs
- Response rate/time, gig quality + count, reviews, active gigs. Top fix is usually: improve the
  weakest gig thumbnail or response time.

## Execute (after approval)
- Message/offer: open thread → paste approved text (or build the custom offer Aise approved) →
  send → `cc_mark_sent`.
- Never send a custom offer with pricing Aise hasn't approved. Delivering order files = Aise.
